If you want to write unit tests on JS classes, sensibly, you need a test framework. Unit testing quite alot of client-side JS code is hard, as it is quite involved with the DOM or webrowser. I present a range of options. Please remember 'does it feel intuitive' isn't something that can be put in a unit test. I am not presenting my own code here, as the current range of tools seem adequate (I have made a few tools previously, but those projects are firmly one of several companies IP. The publically available tools should be better looked after and have more investment, so represent a better solution.).
I repeat to be clear, this article is about unit testing JS modules and Components, not behaviour testing webpages. The I am paid for is generally complex enough that one needs to test each one, to reduce later stage costs. I will write an article on BDD and behaviour testing?.

Metrics I am using for comparing test frameworks:

  • what you can test with X;
  • complexity / time to make stable tests;
  • ability to integrate with a common CI tool;
  • ability to output TAP 1 (for linking into other tools). There is a TAP package in npm 2 3;
  • fragility of tests;
  • if tests are inside the browser, or outside

Writing easy-to-test JS is more complex than other languages. Nicely written Objects ~ JS style ~ do not export many symbols. Most in-browser code is written a pluginable libraries; and each library must allow that there are other random libraries in use at the same time. All the in-browser code shares a single namespace per page. It is possible to achieve fake Namespaces, by creating your object as a child object e.g. com.myCorporation.UsefulLibrary won't clash with UsefulLibrary or com.MonopolyCorpPLC.UsefulLibrary. Pls see 4 or others for more detail. Technically this isn't a namespace (compared to other languages). Another approach is to use SEAF / IIFE ~ there are many abbreviations to this effect ~ more detail at 5 6 or 7. After the widespread adoption of jQuery 8, the second approach is really common.

JS test tools

Writing easy-to-test JS is more complex than other languages, and reliable unit-tests are therefore harder. Closures and Async code mean its harder to see outcomes of a method call. As some people do crypto in JS, unit testing is essential. For BDD and behaviour testing? please read the more relevant article. According to 9 the best CI projects for JS modules are Travis and Jenkins. Most unit test tools can run an entire suit of tests from one config file, and so can be triggered like this. As a different idea, there are commit hooks to your version control.

  • Jasmine is a unit-tester, similar to any other xUnit tool. It is available from git with docs. It runs in-browser, or a pretend browser such as gulp; and has bindings for Node, Python and Ruby projects. TODO: add demo.
    • Strengths: nice shiny xUnit interface. Can be run in a CI platform 10 11 12. It is designed to be easy to read 13 , and borrows ideas from rspec, and related tools. As part of BDD rather than TDD, it defines the setup()/teardown() pair as beforeEach() and afterEach(); but ignoring the words this has the same effect. Jasmine can be extended to support TAP with 14
    • Weaknesses: I think the Jasmine scripts are quite long, apparently so do other developers as there are several script generators eg. 15. Early versions of Jasmine has poor support for async events (which are really common in JS). However this is fixed by now 16.
  • Facebook inc use Jest git as a test tool (install via npm/ yarn). I think this is mostly used for BDD. UPDATE 2021, I have now used Jest, Jest can be run as a CLI tool or in the browser. If you have written complex Jest test, it supports a debugger on the test 17; however this is a bad code smell.
    • Strengths: It can be setup to imitate Mocha/ Jasmine for zero-cost rollout 18. It has a bigger range of primitives than Mocha. As it is du jour, it has many plugins/ CI integration extensions (see npm). It has better integration with Typescript than Mocha. This package jest-runner-tap gives you TAP output, but not on the jest tests. It can test function results, or side effects (i.e. DOM state).
    • Weaknesses: It takes me longer to setup and get it to not crash; other peoples thoughts 19. Not specific to this tool, but Promises need to be managed for tests to pass 20
  • Qunit ~ demo code 21 ~ A small and light unit tester, for JS only (i.e. it can't test CSS or HTML). I use this for my published jquery modules.
    • Strengths: Simple and concise API. As it is just 2 files, it is light in the project repo. If you are demoing that a JS unit is correct, without concern for a large test suite, it is very nimble. Good for tests that need to be run a part of the DOM (i.e. you want to see that a particular DOM element is set to a value). Can be combined with swarm project to run on all webrowsers at once. Can be made to output TAP, via qunit-tap. Can test function results, or side effects (i.e. DOM state).
    • Weaknesses: This needs to be run in a webrowser. It will output DOM manipulations, as test results. As such it is less use for integration with a CI/CD tool, like Jenkins. No test ability to test CSS or HTML.
  • Behat/mink/sahi stack ~ this is same idea as Selenium. I associate it with Symfony, and Behat is written in PHP. Behat implements the same Domain language as Cucumber. The stack can wrap Selenium2 as a second tech implementation. Everzet has published some introductory docs for mink 22. These state that one may run JS from inside a PHP test. The examples that are supplied aren't very useful, but would be easily reproducible. As this is also a full browser, you can do more complex testing. Mink is an abstraction layer, to allow more testing tools to be used; Sahi 23 is a Java & JS page automator, like Selenium1. Sahi acts as an active HTTP proxy.
    • Strengths: With respect to testing JS, unit tests can test everything that appears in a browser. Like Selenium, test conditions can be applied against HTML, CSS 24 and JS state 25.
    • Weaknesses: If you are one of those people that dislike PHP, its PHP. The API for JS is less clean, as your basic test is in PHP. As the test is not native to JS, the event or async stuff is messy. I can't find a integration between Behat (the test schedular tool) and any TAP. This is lots of different bits of technology, so tests are more likely to break than a simpler architecture (four languages when using Sahi, PHP for Behat & Mink, Java and JS for Sahi, and C++/C and JS for the browser).
  • Mocha/phantomjs stack, has a full headless-browser browser implemented in JS git. Mocha is another BDD spec tool 26, written in JS. Moncha is pretty much the same as every other spec tool, see 27, as example. There is phantomjs documentation. As a webrowser, phantomjs isn't a test tool; there are extensions git git 28 . Phantomjs does have evaluate(), which is functionally eval(). Twitter use Phantomjs with Qunit 29 as a another decision making API.
    • Strengths: Mocha supports TAP 30. Phantomjs has close match to Chrome, Safari and a few minor browsers, as it links webkit. As a node project, Mocha links to common CI tools eg 31. For a pure JS unit (no DOM, not much DI), it would be faster to omit phantom i.e. 32
    • Weakness: Unfortunately, this is old webkit, in the fake-browser. Some people (in particular, the Protractor team 33) discourage use of Phantomjs.
