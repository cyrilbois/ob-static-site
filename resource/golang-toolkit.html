I'm starting to think I should add a snoring backtrack to these pages. I do not have a “project starter” project / scripts for Golang the way I do for TS / JS projects. So notes are here, as plan B...

Language target

In June 2024, the current edition of Golang, fresh from the factory is 1.22. As my dev desktop happens to be freshly updated, I can install this with little effort. In four years this has climbed from Golang 1.14. If the expected use includes the HTTP protocol or Crypto, it seems important to use the newest version, as these are having many security changes in the last few months.
Referencing C++ with UTF-8 and POSIX & win32 API; If I add a thread library to that problem spec; this is the sweet spot for Golang. I definitely have a strong opinion that using Golang 1.15+ is desirable by any means needed. This is when they published the complete better module spec, so code linking and loading is much less clumsy.

Tools
  • An IDE: Goland (sorry Jetbrains ad again), VScode works with a plugin 1, The go.dev site recommends vim 2, with 3 4.
    • Add IDE plugins...
  • An online “playground” ~ as a compiled lang, there is no REPL ~ self-described better playground, normal playground, transpile playground, web-assembly playground, git.
  • Package discovery via pkg tool.
  • Package installation via supplied tools `go get` and `go mod` 5 6. For an external tool written in Golang, it can be installed with `go install` 7
Common process
  • I don't know if there will be a shake down in future, or if actual free-standing OSS Golang is rare. I see many corporate vendors, and no github account for a project (regardless of sponsors) which is what often happens in JS/TS/CSS projects.
  • unit test: docs, longer docs docs packages testing, testify, or specialised ones httptest, the cavalry, Gont and more.
  • lint: I think Golangci-lint git docs is everything a project could need/want (look at included linters).
  • docs generator: docs docs package 8 (a standard library). Golang can be used with Swagger docs ~ the author discommended swagger-codegen project, due to language differences.
  • Compilers: builtin, `go build` 9. A page of details for managing building golang-notes
  • The entire language is supporting packages of packages, so no extra tooling for this is needed.
