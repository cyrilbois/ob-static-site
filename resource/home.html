This is a simple site with alot of words. My objective is communication.
When I don't have large projects going on, I am adding more content (and when I do, I am scribbling notes to publish later).

This site is to discuss and describe software architecture and systems, along with practical usage from my hands-on experience of various tools and systems. My emphasis with this space is on engineering content presented with engineering English. Every article on my website has been thoroughly researched. All longer articles contain from 70 to 150 references, I read over 500 specific SERPs, and the longer texts take a month each to write.
Hope you find the reading useful and helpful but I do not consider my articles to be a blog.

About the site

Please note, there are many bloggers who write evergreen articles. I do not understand how this works in detail. If you blog about a recent victory for your local football team: it is news for a week; if well written, a current reference for six months/ a year; after that, its just a webpage. Your team will have changed squad, division, manager, maybe location. If those changes don't make your article irrelevant its not useful to read to start with ( . . . it could be entertaining).

I am tagging creation years on many articles, as I think its more useful to know the context; and periodically I add a new article on the same topic.

The site is:

  1. Assessments of various technology and processes.
  2. Lots of material removed from my CV, so I can keep that concise.
  3. Project notes for various bits of software I have created.
  4. Mostly, alot of words; with lots of bibliography

  • One of the section headings on this homepage was “everyone is as good as their sources”. I removed that section; however you can see all my sources (all those Harvard style reference links, all of them. Mean average of 30 or-so per article).

  • In 2017, I added the create and edit date meta data. The last edit date feature is less useful than I would like (in 2020, then again end of 2022); as I needed to edit every single existing article to add more meta data.
  • In 2018 I put articles in article groups to improve discoverability of content.
  • If I had spent one more day on site infrastructure in 2013, this would have been a JAMstack site since then. UPDATE 2022: my webhost was bought; so I have done this.
  • In the end of 2022, I updated many references to be HTTPS.
  • In 2012 I made the site mobile capable (template v3); but I wasn't expecting anyone to use it from a phone. I put text in columns for readability, like newspapers. I added “page chunks” to the articles so users didn't need to scroll up and down all the time (template v4). I added an RSS, and a site-map in that update as the content had increased. Then 2022, I updated the HTML to a more modern version, with better accessibility (template v6).
  • I am replacing the 2010 and 2012 era CSS with a BEM CSS design; again so I can improve adaptability to different peoples preferences and machines. My palette is now 9 colours, rather than erm, about 50. I will tidyup the font declarations next.
  • I have dropped my “manual mouse hover” code (JS used for triggering adjacent elements), as CSS v4 is now widely enough supported.
  • In order to be semantic, I have standardised that code samples are wrapped in "<CODE lang=“XXX”><PRE class=“code " lang=“XXX”>" blocks. The HTML spec says that PRE isn't for programming languages, but using PRE makes it much more readable.
  • I am tagging older articles with a warning in the strap-line area. I would like to add an URL map (the Apache feature) to pull in likely misspellings, and to make everything lower case
  • After 2012, many of the articles are where I needed to be a local expert in XYZ in a hurry. I am trying to be planned and organised about adding more content. I have deleted 20% of my content that I do not think is useful today.
  • You will note all the github links magically get the Octocat logo. I also decorated the docs links. I do not have a NPM logo to repeat the same trick.
