Objective: deliver better value on this article than graphql.org.

GraphQL is:

  • A spec 1, like REST
  • A uniform way to access real API, and DB (like an ORM), with less code. A sales pitch is “to simplify integration to multiple existing API”
  • In terms of architecture, a library that supplies client and server side; operated via a schema. The client request is sent to the GraphQL server as a single unit; and forms a composite query. Can glue things together 2
  • Better argument against REST: REST designed in 2000 doesn't meet current market; rich apps are complex; faster iterations than jigging REST and client as separate things are desirable
  • GraphQL server uses “type” that look like typescript interfaces, but use a '!' for required; 'Schema definition language' 3 4. Logically, a legacy XML API (i.e. code from 2000-2010) could be ported straight to GraphQL transparently, as SOAP and GraphQL offer similar features.
  • JS reference implementation git
  • Basic contract: Query 5, Mutation 6, and Subscription 7.
  • Server-side limiting features 8: Timeouts 9, Maximum Query Depth, Complexity 10, Throttling 11. These are the same algorithms used in a plain RDBMS, tuning values can be pulled from other platform docs.

In 2018, GraphQL was badly advertised. Quite a lot of those adverts were to the effect of “you don't have to learn to use SQL 'join' if you replace everything with GraphQL, as this will do it for you”; it is not advisable to ask people who cannot use SQL to create API over an RDBMS.

Advantages / benefits

These points are a detailed summary from 12 13 14 15

  • Good fit for complex systems and micro-services; fetching composite data with a single API call.
  • GraphQL people say API wrapping has faster performance and less overheads not sure if dev performance or code performance.
  • No over- and under-fetching problems. Not a problem I have seen with REST, seems a strong feature for FOAF datasets (so would need many joins in an RDBMS).
  • Client writes requests inside a preset schema, and can also transform data. Where the data comes from a local DB, this is useful.
  • Included features: Validation and type checking, auto-generating API documentation, API evolution without versioning, Code-sharing, Permissions are all 'out-of-the-box'. If these are faster to build via schema than server framework tooling, this is good value. However these features are not unique to GraphQL. Business value: Can hire less good developers, as these features are performed by the library, rather than the staff created code.
  • Rapid application prototyping is easy. A strong point for agile teams with uncertain goals/ aims
  • Should be less legacy XML for developers to deal with.
  • Can 'hotswap' database structures without the client noticing (e.g. Postgres cluster to Redis); this is also true for a REST API.
  • PubSub pattern “for free”/ no dev cost, as its a GraphQL library feature.

Negatives/ liabilities

  • Performance issues with complex queries. There's always performance issues with complex queries: as ever, no change
  • Web caching is more complex as all GraphQL requests return HTTP200. Can be managed by creating UUID 16 on the server.
  • Overkill for small applications. This was built for Facebook; a library doesn't need to be deployed in every situation.
  • File uploading isn't supported (no selfie feature); so something like an AWS S3 integration would need to remain.
  • Developers need to be trained/ upgraded, and this takes time. The query language is like elasticsearch, I don't think that learning another language is a big problem so this point should be last.

Reads (query/aggregates)

I have not done similar listing of query structures for any other DB discussed on the website; but I had operational use of the each DB first. Secondly note that many RDBMS support the same common grammar, SQL98 or greater. GraphQL is its own spec and project. I think this section is very boring, and I wrote it.

Simple list of concepts for queries:

  • These are equivalent to SQL select...
  • The API request/ response are JSON dictionaries 1. The request lists items to return; and the response has a “data” key, then the requested fields if present. This resembles MongoDB.
  • The queries can include calls to “functions” (this is what the syntax looks like), these request to pull data from a specific “object” (commonly a database table). The “calls” can take arguments, which are a notation matching SQL 'where clauses'.
  • The docs outline subqueries 2, which they call aliases.
  • In order to reduce required typing on big queries, GraphQL has a fragment structure 3, which resembles a typescript interface; this doesn't map to SQL structure that I can think of.
  • The Fragments can be decomposed, known as “inline fragments” 4.
  • Generally a useful query would be used like a prepared statement, and have a name i.e. start with “query $name”; GraphQL calls this operation name 5. Obviously the query type could be one of other supported actions, which are listed in the first section.
  • Secondly, the operation and name thing supports arguments/ variables 6 7 (I think they used a different name here to make it easier for casual readers to keep track). The use of variables resembles calling the named operation as a function (see docs for this item); this is extended as variables may have default values. It is also possible to use “search” as the verb 8, and get the “object type”, which they call __typename.
  • The docs note that there is locking in effect 9, exactly same as RDBMS.
  • The “function bodies” support branching directives @include and @skip 10

Note: Confused readers may gain from the text in the canonical docs, in the first link in each line, that have a lot of simple examples.


Writes (mutations etc)

This is focussed on mutations i.e. a map of SQL insert, update and delete operations; types, to enforce that expected data is present (same as SQL schema); and validation rules, which would map to SQL constraints, triggers and storproc, and REST behaviours. If a dev has somehow skipped past Typescript, static typing in GraphQL will come as a shock.

  • These match an SQL update operation, and are described 1.
  • As noted in the first section, GraphQL DSL has a type keyword, and this is used to define Typescript interfaces. It uses '!' to indicate required status; see example in 2. The notes say do not assume people are using JS on the server, but the DSL is very similar to JS/ TS.
  • GraphQL DSL supports Enum type 3
  • GraphQL has a schema keyword 4, which is used to define “Objects” used in the queries.
  • GraphQL DSL supports arrays/ lists 5 and supports a useful amount of 'null controls', i.e. process to regulate absent data (same section of docs).
  • On the server side, GraphQL, seems to have interfaces, which behave like Typescript interfaces 6; confusingly Types may extend them ?? and union types 7, used when there is no common data between options.
  • For data structures that are used as “form posts”, the keyword input is very relevant 8. Input-side data cannot have arguments (as far as they are SQL where clauses, this wouldn't make sense).
  • Fragments are like a interface 9, but used as a smaller reusable thing that is linked into larger structures. The usage is very close to traits in OO. Fragments may not be recursively defined, or have a closed definition graph.
  • There are limits on what data can be requested; I think I will need to write myself some unit tests on that, as the manual isn't clear. It says you can't invent a single char field from a string field; but I would build that outcome as a decorator, or a transform using a boutique SQL index.
  • The DSL does support inline fragments, that look like PHP or Python 'with clauses'.

Server implementation

This is what code that needs to be present to supply data requests 1; quelle surprise this looks like a REST API in a modern framework. Notes in the official manual in this section are much shorter.

  • As a system based on the JS engine, it has native support for asynchronous data streams.
  • The manual explains the process of resolving the query/ mutation; after validation is execution...
  • The library supports translating DB enums to “casual normie strings”, but doesn't describe how.
  • The library supports introspection features, which return data like how SQL explain does 2
  • All magic tricks in GraphQL start with a '__', similar to C. Accepted values are __Schema, __Type, __TypeKind, __Field, __InputValue, __EnumValue, __Directive
  • Thorough use of introspection fields gives you a resultset that looks like a schema or a XXX XML description file.
