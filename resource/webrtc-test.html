Web RTC is a JS feature to allow control on video/ audio. It is conceptually near VoiP, as all the notes mention real-time 1 frequently. It assumes a standard internet connection, but is designed for peer to peer architecture. A server is necessary to broker the initial connection, but this is just a common meeting point. The wiki page mention IoT 2, but most of the IoT devices I see are too low CPU power for video encoding. The spec was first published in 2011, but achieved W3C stable status in 2021.
One of the design principals was to hide unnecessary data from the JS interpreter, so peoples machines can't be profiled at scale. An RTC stream is started with a SDP 3, which feels like SIP.
It is suggested to import a compatibility layer/ HAL to harmonise the different JS implementations, such as git npm.

As ever, try to import before locally create; therefore try an RTC library from:

  • Mediasoup git npm docs [released via ISC licence] demo has a number of goals, but mostly broad compatibility other suppliers. They want to be a SFU (Selective Forwarding Unit). Project includes server code, in Rust and client side in JS (ES6).
  • Record RTC git npm docs [released via MIT licence] demo is a RTC client that receives data and stores it.
  • JS SIP git npm docs [released under MIT licence] demo is an implementation for SIP (discovery protocol for VoiP), but bundles in RTC and IM features.
  • This article isn't about lists of libraries, but the technology...
  • JS features MediaDevices MediaStream MediaStreamTrack MediaTrackSettings.
  • Some sample code 4
  • Exposed features where software can control settings: tracks (see MediaStreamTrack) can be paused via the enabled flag; creating a gainNode from AudioContext.createGain allows setting a adaptor on the volume example 5; Video streams can be halted with the stop() call, but must obey temporal constraints.
  • Realtime streaming has limited time-shift capacity (i.e. skip forwards/ skip backwards), for reasons I hope are obvious. Video 6 settings for non-RTC videos are simple and get confused in with RTC search results. Getting the correct screen size is done by “device” selection (this includes current rendered size of a browser tab). I haven't been able to find a screen-size scaling feature, but I can see this being a high value interconnect service, which is present in other AV services (for example watching Netflix on your phone then your home-cinema desktop).
  • Note since the 2000s there was a feature of IP6 called multicast 7, see RFCs in top of link. There are some streaming solutions that use multicast, and if all the switches have been upgraded, can easily support an audience of thousands efficiently. IP6 is being rolled out VERY slowly in the middle layer infra (...its 25-30Y later). WebRTC doesn't seem to use this for scaling, unlike VoiP.
