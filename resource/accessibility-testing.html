This is a specific article about testing for accessibility. Please see other articles on this site for other aspects of testing, easily accessed via testing-epic. In my experience, directors are less interested in funding the work-time to do accessibility testing unless they are targeting sales to a broad end-user market (literally, they do not think there will be any outcomes from that time and money spent). Personally, I think it is a hygiene factor that should be automated as much as possible. Thus the following tools...
Note ~ unlike other types of testing, lack of accessibility can have legal impacts.

Technical definition

A very common blog tactic is to define many words. In this case “what should accessibility tests cover” is critical for compliance, and seems to be “implement the WCAG specs”, there are several releases to this. There are three grades of strictness for compliance: “A”, “AA” and “AAA” 1, although no-one has defined the term, I assume the A is short for accessibility. Requirements for these levels are discussed in 2 3.
Matching the rest of W3C publications, they have numbered edition labels, WCAG 2.0 released 2008, WCAG 2.1 released 2018 and a draft WCAG 2.2 published Sept 2022. Newer versions are more relevant to modern problems, I'm entirely ignoring WCAG1, but I think everyone else does also. The WCAG defines four design principals, to show their document is complete. A website should be: perceivable, operable, understandable, and robust [US english, but it works]; which roughly maps to each of the functional activities that a human needs to do to read something inside a data system. The US government uses a slight simplification 4. To read more about specific fonts, please read readable-fonts.

Boring sidebar: I added a HTML strictness tag to my boring website in 2010, as I was trying to show that I was using HTML correctly. I think semantic HTML is a valuable non-functional objective, as it makes it easier to use the HTML for any other work. Wordpress doesn't make this claim, we are different dev teams. Between Q4 2022 and Q2 2023 I redeveloped up-to site template v4; I have verified and updated all accessibility compliance for today, I need to update the compliance tag. I am now thinking I should remove the HTML correctness and WCAG tag from the non-index pages as its not globally relevant.

What are the EU legal requirements?

After June 2022, EU states must enshrine European Accessibility Act aka “EAA”, which covers access to services be it Cashpoints, taxis, sports on your phone or a random e-comnmerce website/ app. Quoting 5 “The EAA was derived from the UN Convention on the Rights of Persons with Disabilities (UNCRPD).", the UNCRPD can be accessed as 6. The full text for EAA can be accessed via 7 and replaces the older Web Accessibility Directive aka WAD 8. There is an official cover page 9 with access to translations and summaries. Similar cover page for WAD 10. The EU states this may incur a transition cost, but it is beneficial as: you have a broader market including 135 million disabled or old people in the EU which is about 30%; the single standard makes cross EU state trade easier and faster; in future lower operational costs. Wiki claim EAA is in effect from 2019 11, but it is a large scale process to apply to the whole of Europe. Another acronym is European Harmonized Accessibility Standards aka EHAS 12 13 but this seems to be officialese for supporting WCAG.
From the perspective of webapps, supporting Web Content Accessibility Guidelines aka WCAG spec 14 15 is a strong step to being correct for EEA. The second link is a detailed breakdown of suggested behaviours. As EEA is still being rolled out across EU states, and being defined in practical terms, the best practice is still being defined. It is nice when W3C can agree with EU governance. Some more targeted notes are EU vs US, targeting older users and accessible design.
Note ~ the EAA has a carve-out for tiny organisations of less than €2M/Y or less than 10 staff; so can ignore EAA if needed.

What are the UK legal requirements in 2023?

I'm expecting there will be a cascade of changes as a side effect of the politicians not needing to follow the EU.
In the UK, the legal requirements are the UK Equality Act aka UEA 16 17. Concurrently the Equality Act 2010 18 or the Disability Discrimination Act (DDA) 19 is quoted as still being in-effect. Some guidance rather than laws is published as 20 21 22
Similar to the EAA, the UEA maps WCAG guidance. I would guess that the same benefits apply to the UK implementation of WCAG, although the number of disabled / elderly would be smaller. I see a quote of 20% with at least 1 limitation in several places (in previous links).
There are many documents on creating accessible content inside the gov.uk structure, for example 23 24 but that is off topic.



Practical solutions for testing accessibility

It is not practical to make an accessibility QA service for yourself. The following rentable services exist:

  • Primary suggestions:
  • Pagespeed ~ a free page checker from google; it runs in “mostly real-time” (i.e. each page report takes 15-20s). It supports a good range of WCAG, but I'm not sure if it is 100% of it. If you use Analytics, it will merge in the results. It will show you a screen-grab of what each device looks like (from a chrome headless instance). This is the option I currently use. Quite a few features from this tool are available in Chrome as the “lighthouse” tab locally; and according to 1 there is a version that can be run on all URLs in your site at the same time. A benefit of server based test tools that are not running on your local webserver, is that site performance can be measured; this is something that Pagespeed does well.
  • aXe acquired from git or npm can be linked into standard unit tests. There is a paid service that performs all the manual vetting available from Deque in addition. If you had linked this to your test infra, the results would be very convenient. They say they need a custom test URL, which duplicates the normal page, and links the additional aXe library. For render-once sites like blogs, 1 test harness would cover the entire page; but for SPA with locally generated content aXe will need to be triggered after every “human input event” as the DOM will have been changed. They state they have some capacity for JSDOM, running from NodeJS, but its not as reliable.
  • Tenon is an API based service, and seems to mostly focus on HTML. It has spider capacity; it strongly dislikes developers using A elements as buttons. I saw no CSS issues via this, possibly as it wants to do staged processing. Please see level access as they bought Tenon.
  • QualityLogic ~ a rentable service operated by visually impaired software engineers. If they do revisions rather than just testing, they will need a while per client. They do not quote a price in a place I can find. This is a very US centric.
  • QASource ~ rentable service, uses some manual testing, but mostly ML and AL to test. They directly claim that every client is different, and so cannot give general quotes. This is a very US centric.
  • If you a looking at a template/comp/wireframe, using Visbug FireFox chrome is a nice dev experience 2. It doesn't test your site, it gives you a good look at your current design so you can edit it now (this includes editing-tools).
  • Less targeted tools:
  • WAVE ~ a client side only testing browser plugin, run by webAIM. Running since 2001 from the Utah state University; much of this tools marketing is targeting older web trends. Note as a browser plugin this can be run inside firewalls or security hobbles, so it is worth remembering as a backup test tool. UPDATE I tried installing this, it does give reports, but it is very hard to work out which HTML element it is complaining about. Also it doesn't read the whole of the CSS, so some of the objections are are wrong. Thirdly, it doesn't assume that people with limited vision could have adapted their entire machine to have a larger font-size 3 4; so every site has to do it for them separately (my font-sizes are relative, so if you have limited focusing ability you are still supported). Fourthly, it reports elements which are display:none as being small/ poorly coloured.
  • JAWS ~ this is an established text to speech tool that sits on-top of webpages, or ms Word etc; also supports Braille. They currently have a video talking about 20year history, I know this, as I met them doing my second degree. This should be used for 1 user journey “supporting blind people, who are using common blind-support tooling” as it is the most widely used website reader AFAIK.
  • SiteImprove ~ a service that crawls every 5days. SiteImprove is a member of w3c. I cannot find any pricing. Platform also supports Deutsche.
  • SortSite ~ this is a site test tool, not an accessibility tester, but it does check WCAG v2 and v2.1. It has a whole site UI, so if you make each page separately this would save staff time to test, but the smallest licence is 100USD 5.
  • The W3C hosts a very long list of accessibility checkers 6 including ones that are quite old. I think many of these are rules systems, and no fees; but do not have a practical means to check. If you use a Linux, you probably have tidy, which is a convenient HTML validator, and supports WCAG checks. As a HTML validator, it doesn't look at CSS so this is only useful for some situations.
  • Boring sidebar: On my personal site, my PageSpeed scores are all over 95%; this should be achievable on most sites/platforms/apps. The updates didn't cost very much, and can be done as a rolling process on each completed feature. It is the law. The UK government recommends making an accessibility statement 7.
