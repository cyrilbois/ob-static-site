I am writing this to show recent experience with a subject that I did cover in my degrees, but those where a very long time ago. As the sole frontend developer, I literally built or imported all the code; and applied the following UX thinking at the same time. This article is a UX specific extension to UX-engineer, please read it first for context.

High level process

  • Build several Persona, after meetings with relevant people (these are in the git repo). Generically I called them “admin” to set the tone.
  • Take objectives from the PO, as he has best analysis on the legal requirements (also has done a lot of maths analysis, as he has a practical physics PhD).
  • Please note, admin are motivated to be correct/ accountable, but have little interest in the platform. Nothing with serendipity or “cute” is relevant to this platform. Further to this angle, standard marketing things like opinion/tone/feedback gathering isn't relevant.
  • The other members of staff are not UX people. Spend time focussing product suggestions on things that users will benefit from.
  • Work to standardise terminology towards more user centric wording, and secondly to be consistent across the user interface (including external docs) with the legal requirements. We feel that this will make life easier when negotiating with external auditors and solicitors, as the platform is a compliance tool, its data may be used to defend the client in a court case.

Practical details

  • Added consistent and standard iconography.
  • Encouraged a “standard banking colour-scheme”. Whilst all the website technology is my responsibility; other co-lateral of the company isn't. Logo rebrand in early 2018 is now more consistent with other graphical items.
  • Justified, defined as user-stories, then created and deployed standard UI features like login/ logout, which the PO and other stakeholders omitted (I was told several times that meta data on trades is as valuable as NSA meta data on telephony). Made the admins session last as long as their activity did.
  • The lists of servers are sorted in a “most likely to need admin attention” - first order. Defining this is an important user-centric point and non-trivial.
  • With sales team, make original performance categorisation more relevant to admin. The new categories reflect admins concerns (with regard to legal compliance), so the platform is much faster to quickly browse. Performed a few timed scenarios to test this change.
  • In early sprints did not have time/budget to perform proper software engineering practice; but these where exploratory. Quickly built user journeys to allow visualisation of a fairly abstract material. Since then, the product definition has iterated, and we removed these.
  • As users are not going to be impressed by anything they routinely build for themselves in powerpoint, did not animate the graphs. Did reduce graph loading time. My analysis is “this will always be a boring reporting task, so a quick boring task is better”.
  • I built the first version of the monthly PDF compliance reporting to be mailed to users, to reduce manual work for them.
  • The product has alerts for atypical events that IT support may need to manage. Iterate the templates for these technical notices so they work on small form phone mail apps more easily. The most frequent user of these alerts is much happier with new version.
  • Re-skin notifications, so they are easier to read and have a higher/more useful data:noise ratio.
  • Although very atypical for UX, I have accumulatively spent months on making the data smoothed/ filtered/ averaged/ axis clipped, as admin need to be able to get a clear picture of their network. Without this computation ~ to make the data less like noise ~ the raw numbers are fairly useless.
  • Focus development budget on devices that admin are likely to use, setup feedback on devices in use; to ensure that the experience is efficient.
  • Add “pages” of results, in order to improve “useful data” accessibility (i.e. the few percent of the dataset that admin need to read and act on, today). This feature is like a standard e-commerce website. In most cases this reduces the amount of irrelevant data that people need to scroll though by a large percentage.
