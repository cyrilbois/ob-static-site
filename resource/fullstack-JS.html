I am an experienced software engineer, and have built things with JS. I have used a wide range of JS frameworks; but can also write in raw JS. My experience as a JS based developer is:

  • All my code is “use strict”. Due to writing in other languages before JS, I rarely hoist;
  • I would like to use ES6 but don't think that there is enough support in browsers (this is improving 1). As of late 2016, I am writing in es5 for code in webbrowers. I still need to import a few polyfill functions for MSIE. Update late 2018: most of ES6 and ES7 works in evergreen browsers, but tools to dynamically add compatibility layers for MSIE and Safari are still commercially important;
  • I went to a JS conference in 2020, see notes;
  • UPDATE 2022, I have now used typescript. I think the type system in Golang is better; but I see the commercial value in using typescript for projects, in the browser or a server side API.
  • I have used Node commercially, in 2015, for a particular project I was extending the functionality, the first version was built with Mongoose and Express. Koa would be a better framework IMO; as generators lead to less nested callbacks, and secondly it has less of a maze of extensions. That project also used handlebars, and the JS templating library (bra/pug);
  • If I was starting a fresh project, I would look at Deno, for serverside JS; as its easier to manage deps.
  • My most recent project uses Vue2. This is mostly complete, I am the sole developer. I have built a side-project using Vue3 shopping-list;
  • I worked for over a year on a commercial product using Reactjs (the same company). I made major changes to a second project in “React with hooks”, reactnative-developer;
  • I worked for over a year on a commercial product using Angularjs at lead-engineer;
  • As most people would aggregate UX with client side work, I refer to some case studies UX and more UX, and third.
  • I prefer to write in objects/modules, it makes testing much easier and cheaper. I can write code towards OO or towards FP architectural styles;
  • I am currently unit testing with Mocha/Chai, for example. Historically I used qunit, as it is a really small dependency, but this wouldn't scale for CI style use;
  • I do functional testing, historically with Selenuim, more recently the same Mocha/Chai;
  • I use closures; My libraries tidily don't pollute the symbolspace anymore than is necessary. I know what the self executing function declarations are;
  • I currently don't use any JS specific design patterns, historically jQuery patterns; but as an OO engineer, I use the following. Where I am working on a mostly JS project for a while, my code will be OO SOLID.
  • Focussing on reports, I really like datepickers; its removes two days off any isolated platform development. UPDATE: Since I started saying this, browsers now support a richer range of HTML5 form items, secondly for new apps, I normally use a JS date picker with a clock picker;
  • I have built SPA. For current technology, I would be using a JS templating and content injection tools, for example Vue2 + Vuex.
  • General consensus is that data manipulation should be on the server, so it can't be disabled. When I build time limited API, I send time as an Epoch int to the server;
  • I have worked in high volume services/sites, but the server language isn't JS;
  • I have done some demo code with websockets, but as fundamentally its a TCP socket, this is no different to normal POSIX programming. I have written alot of normal TCP and UDP based programming;
  • With regards to UI, I target HTML5, mobile and RWD. I use CSS frameworks for cost reasons. My personal site is targeting desktop mostly, due to the volume of words; but there is CSS for mobile;
  • The following are JS heavy projects:
    • reactnative-developer ~ most recent
    • UX-engineer
    • UX verification library in JS.
    • lead-engineer
    • route-plotting
    • visitors
    • ms-projects
    • senior-developer ~ used Netservices in JS
